User Interviews

Interviewing users of our design system

I interviewed 20 designers, developers, and PM’s to better understand users of the system. I asked questions about their thoughts around our existing system, in addition to their wants and needs for our future design system.

Audit Existing System

Understanding our existing design system

In tandem with user interviews, I analyzed the current system to better understand its capabilities, in addition to identifying potential gaps and opportunities.

Key takeaways from system analysis
  • Prevalent inconsistencies between system and storybook
  • Need for typographic scale
  • Opportunities for quick wins with color
  • Need for system governance and process
  • Inconsistencies throughout the system

Audit Presentation

Presenting my findings and recommendations for next steps

Compiling everything I discovered about the current system, I created a presentation highlighting key gaps and opportunities in addition to recommendations for next steps to bridge those gaps.

In an effort to increase awareness around my work and transparency around our design system; I presented to two teams. I presented my detailed findings to the smaller UX team first, then a condensed version to over 70 people from the larger product team.

Recommendations

Recommendations for next steps

I presented two main options for a path forward in our design system.

Path #1 - Update current system

Cons:

  • Any updates to existing components can, and likely will, cause breaking changes to existing designs.
  • Updates will have to work around or with existing components
  • Current library has gaps in component flexibility
  • Does not support all 5 products

Pros:

  • Does not require a new library
Path #2 - Start a 3.0 system, updating and pulling in components from 2.0

Cons:

  • Would need to create a new library file and have designers add new library file
  • Potential issues with designers still using old components

Pros:

  • No effect on old and existing designs
  • Freedom to establish system structure before components
  • Flexible components built from the ground up
  • Can support all 5 products

Post-Audit

Excitement and hesitancy surrounding the unknown

There was overwhelming excitement about the possibility of a 3.0 design system. At the same time, there was significant hesitancy on what that would look like and what an updated system could do.

To address and alleviate these concerns I created a proof of concept using a single component: the button.

I decided on the button for a proof of concept because I wanted to clearly illustrate the power of flexibly built components. With a flexible component, Instead of needing 5 separate button components or variants for each product and brand; a single component combined with variables can be used.

Proof of Concept

Presenting buttons as a proof of concept

Taking my proof of concept button, I again presented to the same two product teams.

The presentation gave designers + developers a chance to ask questions, voice concerns, or comments. Additionally, the presentation surfaced my work and the changes coming with our 3.0 design system.

Following the presentation I was given the green light to build upon my proof of concept and begin Medical Solutions 3.0 design system.
